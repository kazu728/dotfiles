# プロンプト振り返り

過去の自分のプロンプトから**改善点を見つける**。目的は2つだけ。

1. **傾向** — 複数のプロンプトに跨って繰り返している癖を、実例を引用して示す。
2. **対策** — それを繰り返さないための、できるだけ恒久的な手当て。

何も外に送らない。ログは自分のマシンから出ず、分析はこのセッションの中で完結する。

## 手順

### ① 取り出し（決定的）

自分の実プロンプトだけを取り出す。subagent（`isSidechain`）・スラッシュコマンド・システムメッセージ・割り込み・空文字列は除外し、セッションごとに時系列で並べる。既定は直近30日。

```bash
SINCE=$(date -u -v-30d +%Y-%m-%dT%H:%M:%SZ)   # 期間を変えるならここ（-v-7d など）
find ~/.claude/projects -name '*.jsonl' -print0 | xargs -0 cat 2>/dev/null \
| jq -c --arg since "$SINCE" '
    select(.type=="user" and .isSidechain!=true and (.message.content|type=="string"))
    | .message.content as $t
    | select(($t|test("^(<command-name>|<local-command|<command-message>|Caveat:|<system-reminder>|\\[Request interrupted)"))|not)
    | select($t != "" and .timestamp >= $since)
    | {project:((.cwd//"?")|split("/")|.[-2:]|join("/")), session:(.sessionId[0:8]), ts:.timestamp, text:($t[0:1000])}
  ' \
| jq -s 'sort_by(.session, .ts)'
```

- 特定プロジェクトに絞るなら jq に `| select(.cwd|test("リポジトリ名"))` を足す。
- 長い貼り付けは intent を残して 1000 字で切り詰めてある。`session` 単位で並ぶので、各セッションの先頭＝初手（cold start）が分かる。

### ② 分析（ルーブリック）

各プロンプトを次の4軸で見る。ただし**数字を付けるのが目的ではない**。狙いは繰り返す癖の発見。

- **clarity**：要求は曖昧でないか。
- **specificity**：必要な情報があるか。**明示（ファイルパス・エラー文・要件）だけでなく暗黙の文脈（git diff・直近の編集・会話の続き・プロジェクト構造）も含めて**判断する。
- **actionability**：すぐ着手できるか、聞き返しが要るか。
- **scope**：粒度は適切か。

**文脈考慮（最重要）**：短いプロンプトを罰しない。会話の続き（セッション2手目以降）や git 文脈がある短文は良いプロンプト（例：「git commit」「テスト流して」）。痛いのは**初手なのに文脈もなく曖昧**なものだけ（例：「直して」）。目標は全プロンプトを長くすることではなく、Claude が必要なものを持っている状態にすること。

問題は Anthropic のプロンプト指針に対応づけて捉える。

### ③ 出力

1. **対象範囲** — 件数・期間を一行で。
2. **傾向** — 反復している癖を2〜4個。各々、**実際のプロンプトを引用**して根拠を示す。
3. **対策** — 各傾向に紐づけ、恒久策を優先する（下記）。

## 規範（でっち上げを禁じる）

- **引用必須**。実プロンプトを引けない指摘は書かない。
- **反復のみ**。複数プロンプトに跨る癖だけ。一回限りは無視。
- **無ければ無いと言う**。繰り返す問題が見当たらなければ「明確な傾向なし。プロンプトは概ね機能している」と書く。ひねり出さない。低摩擦なのは良い結果。
- **一般論コーチング禁止**。「もっと具体的に」単体は書かない。必ず引用した実例に対する before/after に落とす。

## 対策は恒久策を優先する

「次は気をつける」（意志頼み＝忘れる）ではなく、**繰り返しを仕組みで止める**手当てを先に出す。優先順:

1. **CLAUDE.md 追記** — 毎回同じ文脈/コマンドを書いているなら、それを CLAUDE.md に置く（二度と書かなくて済む）。
2. **skill / command 化** — 毎回同じ手順を指示しているなら、スキルやコマンドにする。
3. **hook / 設定変更** — 毎回同じ後処理・是正をしているなら、フックや settings で機械化する。
4. **言い回し（最後の手段）** — 上記で消せない曖昧さだけ、引用つき before/after を示す。

例：「project X の初手で毎回 build/test コマンドを書いている」→ 弱い対策「書かなくていい」ではなく、**強い対策「X の CLAUDE.md に build/test を書く」**。
