# draft-fable-prompt

Claude Fable 5 に渡すプロンプトを作る。
Codex 自身は、依頼された調査・レビュー・設計を実行しない。

## Reference

まず公式ガイドを読む。

- https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5

ガイドの内容を今回の依頼に必要な範囲で反映する。
ローカル要約やコピーは持たず、公式URLを正とする。

## Prompt Design

ユーザーの依頼を、Fable 5 がそのまま実行できるプロンプトに変換する。

各要素は「これが無いと Fable 5 が具体的に何を間違えるか」を言えるときだけ入れる。言えないものは入れない。長さでなく、全行が load-bearing かで判断する。

候補（デフォルトは入れない。必要判定を通ったものだけ入れる）。

- 目的
- 背景
- 読むべき repo、ファイル、URL、仕様、既存成果物
- 作業範囲
- やってよいこと
- やってはいけないこと
- 必要な subagent 分担
- 成果物ファイル
- 最終回答で伝えること

Fable 5 が repo やガイドから読める規約、標準的なやり方、既知・自明の事柄は書かない。
手順（how）より意図と制約（what/why）を書き、やり方はモデルに選ばせる。
削るときは手順から削り、目的・境界・禁止と、モデルが誤って埋めそうな空白は残す。

成果物ファイルはユーザー指定を優先する。
指定がなければ、案出し・設計は `DRAFT.md`、レビューは `REVIEW.md` を候補にする。

subagent は、独立して調べられる観点がある場合だけ使わせる。
役割はタスクごとに決め、固定テンプレートにはしない。

ユーザーが実装を望んでいない場合は、成果物ファイル以外を変更しないよう明記する。
破壊的操作、環境変更、外部サービスへの書き込み、git 操作は、禁止または確認対象にする。

Fable 5 には、進捗や結論を実際に読んだファイル、仕様、URL、コマンド結果に基づけるよう指示する。
未確認事項は未確認と書かせる。

内部推論や chain-of-thought は出させない。
代わりに、根拠、判断基準、結論、トレードオフを書かせる。

最後は、推奨案、優先順位、次アクションのいずれかに収束させる。

## Output

コピペ可能な Fable 5 向けプロンプトを Markdown のコードブロックで出す。
必要なら、コードブロックの前に設計意図を短く説明する。
