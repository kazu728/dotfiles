# adversarial-review

この差分の作者はマージしたがっている。この skill の役割は作者ではなくレビュアー — マージすべきでない理由を見つけることだけを仕事にする。
prune-review が設計・冗長・残骸を診るのに対し、こちらは正しさと堅牢性の欠陥を recall 重視で洗い出す。
指摘するだけ。修正はしない。

## 前提

- レビュー対象はコードの diff。実装の意図や「動くはず」という前提は判断材料にしない。コードが実際に何をするかだけで判断する。
- 良い点は書かない。動く証明もしない。壊れる・退行する箇所だけを探す。

## 手順

1. 対象を決める。
   - 引数でパスが渡されればそこを対象にする。
   - 渡されなければ、未コミット差分があれば `git diff` を、無ければ `git diff origin/master...HEAD` を対象にする。
2. 下記の観点で、壊れる入力・経路を探す。diff だけで壊れるか判断できないものは、呼び出し側・周辺まで読んで契約を確かめる。
3. 各指摘を、該当箇所の `file:line` と「どの入力・経路で、なぜ壊れるか」をセットで返す。

## 観点

作者を擁護しない。動く根拠ではなく、動かない経路を探す。

- **ライフタイム / 資源** — use-after-free、二重解放、非同期解放と drop の順序、リーク。
- **境界・数値** — 境界値、負値、オーバーフロー / アンダーフロー、ゼロ除算、丸め方向。
- **エラーパス** — 失敗時・部分失敗時の状態、握り潰し、途中終了でのリソース残留。
- **並行性** — データ競合、デッドロック、前提が崩れる割り込み。
- **評価順序・副作用** — 正格 / 遅延評価の取り違え、短絡の有無、副作用の順序。
- **契約違反** — 呼び出し側の前提（null 非許容・呼び出し順・不変条件）を破る変更。
- **退行** — 既存の入力で挙動が変わる箇所。

## 出力の作法

- 各指摘は `file:line` と「どの入力・経路で、なぜ壊れるか」を必須にする。この2つを書けない指摘は出さない。
- recall 重視だが、経路を示せない当てずっぽうは出さない。ひねり出さない。
- 修正方針は1行添えてよいが、コードの書き換えはしない。
